# SecureEnvelopeKit Security Notes (Swift)

These notes cover the Apple (CryptoKit) implementation. The shared,
cross-platform security model is at [`../../../docs/SECURITY.md`](../../../docs/SECURITY.md)
and the normative format is in
[`../../../docs/spec/secure-envelope-v1.md`](../../../docs/spec/secure-envelope-v1.md).

## Security Model

SecureEnvelopeKit protects caller-provided payload bytes with authenticated
encryption. It assumes the caller already has suitable input key material from a
higher-level key agreement, ratchet, or key-management system. The package does
not create identities, establish sessions, rotate ratchets, persist secrets, or
contact remote services.

Public envelope metadata is authenticated but not encrypted. Treat
`keyIdentifier` and `publicContext` as routable public data and do not place
plaintext secrets in either field.

Parsed metadata is authenticated only after `SecureEnvelopeOpener` or
`SecureEnvelopePreview` successfully verifies AES-GCM with the expected key
material. Before that point, a parsed envelope's metadata is an untrusted routing
hint for key lookup, not an authorization or UI trust signal.

## Algorithms

The only supported v1 suite is `v1AES256GCMHKDFSHA256`:

- AES-256-GCM is provided by CryptoKit.
- HKDF-SHA-256 is provided by CryptoKit.
- Random salt and nonce bytes are generated by Security.framework.
- AES-GCM nonces are 12 bytes.
- HKDF salts are 32 bytes.
- Authentication tags are 16 bytes.

The package does not manually implement AES, SHA, HKDF, or random number
generation.

## Envelope Authentication

The binary wire format has fixed field ordering and explicit length prefixes.
The deterministic header bytes, including version, suite, public metadata, salt,
nonce, ciphertext length, and tag length, are passed to AES-GCM as additional
authenticated data. Any header or ciphertext tampering causes opening to fail
with `SecureEnvelopeError.authenticationFailed` after parsing succeeds.

Malformed data, unknown versions, unknown suites, invalid fixed lengths, empty
key identifiers, truncation, and trailing bytes are rejected before decryption.

The stable v1 fixture is stored at the repository root in
`fixtures/SecureEnvelopeV1/secure-envelope-v1.json` and shared across platforms.
Treat any change to that fixture's envelope bytes, authenticated header bytes,
HKDF info, or derived key as a wire-format compatibility change that must be
coordinated across platform packages.

## Preview Guidance

`SecureEnvelopePreview` is limited to caller-provided preview envelopes. Keep
preview payloads small and set `maxPlaintextBytes` for the notification or
display surface. The helper must not be used as a hidden full-message decryptor,
history-sync path, storage lookup path, network path, ratchet advancement path,
or UI renderer.

Preview callers should also bound the serialized envelope, authenticated header,
and public metadata sizes before using untrusted bytes in memory-constrained app
extension paths. The plaintext display limit does not by itself cap public
metadata parsed before decryption.

## Operational Boundaries

Callers are responsible for:

- Generating or receiving at least 32 bytes of strong input key material.
- Keeping key material out of logs, analytics, crash reports, and public
  metadata fields.
- Binding metadata values to the surrounding protocol so public routing data is
  intentional.
- Versioning higher-level protocol changes instead of silently changing v1
  envelope semantics.
- Sharing stable binary fixtures with other platform implementations before
  relying on cross-platform interoperability.
